# Project Governance

## Overview

AegisPCAP is an open source project committed to transparent and inclusive governance.

## Project Structure

### Core Team

The Core Team consists of maintainers who have demonstrated long-term commitment to the project.

**Responsibilities:**
- Review and merge pull requests
- Triage issues
- Make technical decisions
- Mentor contributors
- Maintain project roadmap

### Contributors

Anyone who contributes code, documentation, or other improvements to the project.

**How to Become a Contributor:**
1. Submit a pull request
2. Follow the [Contributing Guide](CONTRIBUTING.md)
3. Participate in code reviews
4. Help with documentation

### Committers

Contributors who have made significant contributions and demonstrated commitment may be invited to become Committers.

**Committer Privileges:**
- Direct commit access to repository
- Vote on technical decisions
- Participate in release planning

**How to Become a Committer:**
- Sustained contributions over 6+ months
- High-quality code and reviews
- Demonstrated understanding of project architecture
- Nomination by existing Core Team member
- Approval by majority vote of Core Team

## Decision Making

### Consensus-Based

We strive for consensus on all decisions. When consensus cannot be reached:

1. **Discussion**: Open discussion on GitHub Issues or Discussions
2. **Proposal**: Formal proposal with rationale
3. **Feedback Period**: Minimum 7 days for community feedback
4. **Vote**: Core Team votes if consensus not reached
5. **Implementation**: Decision is documented and implemented

### Voting

- **Simple Majority**: Most decisions (>50% of Core Team)
- **Supermajority**: Major changes (>66% of Core Team)
- **Unanimous**: Governance changes (100% of Core Team)

## Technical Decisions

### Architecture Changes

Major architectural changes require:
1. RFC (Request for Comments) document
2. Community discussion period (14 days minimum)
3. Core Team approval

### Feature Additions

New features should:
1. Align with project roadmap
2. Have clear use case and requirements
3. Include tests and documentation
4. Pass code review

### Breaking Changes

Breaking changes require:
1. Strong justification
2. Migration guide
3. Deprecation period (1 major version minimum)
4. Community notification

## Release Process

### Versioning

We follow [Semantic Versioning](https://semver.org/):
- **Major**: Breaking changes
- **Minor**: New features (backward compatible)
- **Patch**: Bug fixes

### Release Cycle

- **Major Releases**: Annually
- **Minor Releases**: Quarterly
- **Patch Releases**: As needed

### Release Checklist

1. All tests passing
2. Documentation updated
3. CHANGELOG.md updated
4. Security scan completed
5. Release notes prepared
6. Core Team approval

## Code of Conduct

All participants must follow our [Code of Conduct](CODE_OF_CONDUCT.md).

### Enforcement

Violations will be handled by the Core Team:
1. **Warning**: First offense
2. **Temporary Ban**: Repeated offenses
3. **Permanent Ban**: Severe or continued violations

## Conflict Resolution

Conflicts should be resolved through:
1. Direct communication between parties
2. Mediation by Core Team member
3. Core Team decision if unresolved

## Amendments

This governance document may be amended by:
1. Proposal by any Core Team member
2. Discussion period (30 days minimum)
3. Unanimous approval by Core Team
